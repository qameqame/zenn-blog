## このガイドでやること

LLMを使ったアプリケーション開発を始めると、避けて通れない構成があります。**RAG（Retrieval-Augmented Generation）** です。

LLMは学習済みの知識しか持っていません。自社のドキュメント、最新の仕様書、プロジェクト固有の情報——こうした「モデルが知らないデータ」を扱うには、外部の知識をリアルタイムで検索して文脈に注入する仕組みが必要です。それがRAGです。

本ガイドでは、pgvectorとGeminiを使ってRAGシステムをゼロから実装し、Tool Use・AI Agents・MCPへと発展させていきます。

- Step 1: Embedding・Vector DB・RAGの基礎実装
- Step 2: AIアーキテクト視点での設計判断
- Step 3: Tool Use（LLMが自律的にDBを検索）
- Step 4: AI Agents（複数ツールの組み合わせ）
- Step 5: MCP（ツールをサーバーとして公開）
- Step 6: クラウドデプロイ（Render × Supabase）

## 3つの概念を先に整理する

### Embedding（埋め込み）とは

コンピュータは文字列のままでは「意味の近さ」を計算できません。Embeddingはテキストを数値のリスト（ベクトル）に変換する技術で、意味が近い言葉は似たパターンの数値列になります。

```
「犬」  → [0.82, 0.75, 0.10, ...]  768個の数値
「猫」  → [0.78, 0.72, 0.12, ...]  ← 犬と似たパターン
「銀行」→ [0.08, 0.10, 0.85, ...]  ← 全然違うパターン
```

この変換をGeminiのEmbeddingモデルが担当します。

### Vector DB（ベクトルデータベース）とは

通常のDBはキーワードの一致で検索しますが、Vector DBは「数値の距離」で検索します。

```sql
-- 通常の検索（キーワードが一致しないと見つからない）
SELECT * FROM docs WHERE body LIKE '%F1スコア%';

-- ベクトル検索（意味が近ければキーワードが違っても見つかる）
SELECT * FROM docs ORDER BY embedding <=> クエリのベクトル LIMIT 3;
```

「評価指標の測り方」と検索しても「F1スコアの計算方法」のドキュメントが見つかるのはこのためです。今回はPostgreSQLの拡張機能 **pgvector** を使います。

### RAG（検索拡張生成）とは

LLMは学習時のデータしか知りません。RAGは「検索して取得した情報をLLMに渡す」設計パターンで、モデルが知らないデータでも回答できるようになります。

```
【通常のLLM】  質問 → 学習済み知識のみで回答
【RAG】        質問 → Vector DBで検索 → 結果をLLMに渡す → 文書に基づいて回答
```
